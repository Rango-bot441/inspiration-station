# 创作者灵感加油站

「创作者灵感加油站」是一个面向抖音、小红书等内容创作者的 AI 选题与标题工作台。它聚焦创作者日常最高频、也最容易卡住的环节：不知道今天写什么、不知道标题怎么包装、看见对标内容很火但拆不出方法。

项目当前以单页 Web 应用形式实现，适合快速验证 AI 在内容策划工作流中的落点。

## 解决什么问题

很多创作者并不缺表达欲，真正消耗时间的是选题判断和标题包装：

- 每天反复纠结内容角度，灵感来源不稳定。
- 标题容易写成信息陈述，缺少点击理由。
- 看完对标爆款后，只知道“它很火”，但不知道结构为什么有效。
- 标题测试后的经验没有被记录，下一次仍然从零开始。

这个项目尝试把「生成灵感」「拆解爆款」「记录反馈」串成一个轻量闭环，让创作者更快完成从想法到可发布标题的第一轮草稿。

## 目标用户

- 抖音、小红书、视频号等平台的新手创作者。
- 需要批量产出标题和选题方向的内容运营人员。
- 想复用对标账号表达结构的垂类博主。
- 正在验证内容方向、还没有稳定选题方法的小团队。

## 核心业务洞察

内容创作的低效，往往不是单点能力不足，而是工作流断裂：

1. 选题阶段缺少足够多的可选角度。
2. 标题阶段缺少清晰的点击钩子。
3. 对标学习阶段缺少结构化拆解。
4. 复盘阶段缺少可沉淀的数据反馈。

因此本项目没有只做一个“标题生成器”，而是设计为一个小型创作者工作台：先生成，再分析，再记录反馈。这样 AI 不只是给答案，也帮助用户逐步形成自己的标题偏好和内容判断。

## 产品机制设计

### 1. 标题生成机制

用户输入产品或内容卖点，并选择内容类型。系统会生成 10 个标题，标题生成规则强调三类爆款元素：

- 前 5 秒钩子：反常识、悬念、痛点、利益点。
- 数字或数据：步骤、金额、时间、比例。
- 情绪词：惊讶、避坑、后悔、太绝了等强反馈词。

这套机制的目的不是追求夸张表达，而是帮助创作者快速找到“用户为什么愿意点进来”的表达切口。

### 2. 爆款分析机制

用户可以粘贴一篇对标爆款文案，系统会拆解：

- 开头钩子类型
- 中间逻辑
- 结尾引导
- 可复用句式模板
- 优化建议
- 仿写标题

这让对标不再停留在主观模仿，而是变成可复用的结构提取。

### 3. 反馈记录机制

每个生成标题旁边都有「有用 / 没用」反馈按钮。反馈会写入浏览器本地存储，并在 24 小时统计弹窗里汇总点赞最多的标题和共同特征。

这个机制让用户可以看到自己更偏好的标题结构，例如数字型、避坑型、强情绪型、明确人群型等。

## 核心功能模块

| 模块 | 说明 |
|---|---|
| 爆款标题生成器 | 根据卖点和内容类型生成 10 个可参考标题 |
| 爆款分析器 | 拆解对标文案结构，输出模板、建议和仿写标题 |
| 本地反馈记录 | 使用 localStorage 记录标题点赞 / 点踩 |
| 标题特征统计 | 汇总过去 24 小时点赞标题的关键词和结构特征 |
| 自动备用模型 | 主模型繁忙时自动尝试备用模型，提高可用性 |
| Loading 与错误提示 | 请求中显示加载状态，服务繁忙时给出友好提示 |

## 技术实现

当前版本是纯前端单页应用：

- `index.html`：页面结构、样式和交互逻辑。
- `server.js`：静态页面服务 + AI 请求代理，从环境变量读取模型服务配置。
- `localStorage`：存储标题反馈日志。
- OpenAI-compatible API：调用兼容 Chat Completions 的模型服务。

前端没有使用构建工具，也没有依赖框架。这样可以降低运行门槛，方便快速预览和部署。

## 配置说明

项目不会在仓库中保存真实密钥。运行时由 `server.js` 从环境变量读取模型服务配置，浏览器不会看到真实密钥。

参考 `.env.example`，在本地或部署平台中配置同名环境变量：

```txt
AI_API_BASE_URL=https://your-openai-compatible-endpoint.example.com
AI_API_KEY=your-api-key
AI_MODEL_FALLBACKS=model-a,model-b
```

> 注意：不要把真实密钥写入 `index.html`、`config.example.js` 或 README。生产环境建议始终使用服务端代理。

## 本地运行

方式一：使用内置 Node 服务。

```bash
AI_API_BASE_URL="https://your-openai-compatible-endpoint.example.com" \
AI_API_KEY="your-api-key" \
AI_MODEL_FALLBACKS="model-a,model-b" \
npm start
```

然后访问：

```txt
http://localhost:3000
```

方式二：只预览 UI。

```bash
open index.html
```

直接打开文件可以预览界面，但 AI 请求需要通过 `npm start` 或部署后的后端代理运行。

## 部署到公网

### 静态部署

可以部署到 GitHub Pages、Netlify、Vercel、Cloudflare Pages 等静态托管平台。

静态部署时需要注意：

- 不要把真实密钥提交到仓库。
- 如果使用纯静态托管，需要另外提供后端代理接口。
- 如果使用 Render / Railway / Fly.io 等 Web Service，可以直接使用本项目的 `server.js`。

### 推荐的后端代理方式

生产环境更推荐增加一个轻量 API 代理：

```txt
前端页面 -> 自己的后端代理 -> OpenAI-compatible 模型服务
```

本项目已经内置这个代理结构，前端只请求 `/api/chat`，真实密钥只存在服务端环境变量中。

## 当前版本边界

- 不包含用户账号体系。
- 不包含云端数据库。
- 反馈数据只保存在当前浏览器。
- 标题特征统计使用简单规则提取，不是完整数据分析系统。
- AI 输出质量依赖所配置模型和服务稳定性。
- 公开部署时需要自行处理密钥保护和请求限流。

## 后续迭代方向

- 创作者画像：根据账号阶段、内容领域、目标人群生成不同策略。
- 标题标签体系：自动识别钩子、情绪、数字、人群、利益点。
- 历史复盘面板：展示不同标题结构的长期反馈趋势。
- 爆款样本库：沉淀用户常用对标文案和可复用句式。
- 选题日历：把生成结果组织成一周或一个月的内容计划。
- 后端代理：统一管理密钥、模型路由、限流和日志。
